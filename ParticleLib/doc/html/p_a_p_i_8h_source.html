10\section intro Introduction
11The Particle System API (Application Programmer Interface) allows C++ programs to simulate the dynamics of particles. It is intended for special effects in
12interactive and non-interactive graphics applications, not for scientific simulation, although principles of Newtonian physics have been used to implement the
13particle dynamics where applicable. </p> <p>The API consists of four sets of functions:
14// <ul><li><b>Particle Group Calls</b> - A particle group is a system of particles that are all acted on together. It can be considered a dynamic geometric
15object. From the point of view of the graphics system, a particle group is just another model to be drawn. These calls create and destroy particle groups, etc.
16</li>
17// <li><b>Source State Calls</b> - As in OpenGL, some calls actually do something and some calls modify the behavior of the subsequent calls that do something.
18The source state setting calls set the attributes of particles that will be created with the Source or Vertex calls. The source state calls are in the
19PContextSourceState_t part of the PContextState_t structure. </li>
20// <li><b>Actions</b> - These implement the particle dynamics. Bounce and Gravity are examples of actions. </li>
21// <li><b>Action List Calls</b> - These create and operate on lists of actions.</li></ul>
22
23<p>Following are several basic concepts central to the Particle System API. </p>

25\subsection particles Particles
26<p>For the purposes of the Particle System API, a particle is not just a very tiny speck. Particles can be drawn as anything - water droplets, birds,
27tumbleweeds, boulders, etc.. The Particle System API is generally useful for operating on many similar objects that all move and change according to the same
28basic rules, no matter what the objects and rules are. </p> <p>A particle in the abstract sense used by the API is merely an entity with a small set of
29attributes such as position and color that ultimately dictate the particle's behavior and appearance. Particles are declared in pParticle.h or \ref Particle_t.
30Here are the particle attributes: </p> <table border="1"><tr><td>Position </td><td>location of the particle center </td><td>3 floats </td></tr><tr><td>Color

42\subsection actions Actions
43<p>Actions are the core of the Particle System API. They are the functions in the API that directly manipulate particles in particle groups. They perform
44effects such as gravity, explosions, bouncing, etc. to all particles in the current particle group. A program typically creates and initializes one or more
45particle groups, then during each frame it calls particle actions to animate the particles then gets the particle data and draws the group of particles onto the
46screen. </p> <p>The Particle System API uses a discrete time approximation to all actions. See TimeStep for information on improving simulation quality by
47varying the time step, <i>dt</i>. </p>
48
49\subsection actions Actions
50<p>Actions modify the position, color, velocity, size, age, and other attributes of particles. All actions apply to the current particle group, as set by
51CurrentGroup. Some actions will add particles to or delete them from the particle group, and others will modify the particles in other ways. Typically, a series
52of actions will be applied to each particle group once (or more) per rendered frame. </p> <p>Remember that the amount of effect of an action call depends on the
53time step size, dt, as set by \ref TimeStep.</p> <p>Some functions have parameters with a default value of P_EPS. P_EPS is a very small floating point constant
54that is most often used as the default value of the epsilon parameter to actions whose influence on a particle is relative to the inverse square of its distance
55from something. If that distance is very small, the amount of influence approaches infinity. Since all actions are computed using Euler's method, this can cause
56unsatisfying results in which particles are accelerated way too much. So this epsilon parameter is added to the distance before taking its inverse square, thus
57keeping the acceleration within reasonable limits. By varying epsilon, you specify what is reasonable. Larger epsilon make particles accelerate less. </p> <p><a
58href="ParticleLib/doc/html/class_p_a_p_i_1_1_p_context_actions__t.html">List of Actions</a></p>
59
60\subsection alists Action Lists
61<p>Action lists are blocks of actions that are applied together to a particle group. An action list is conceptually similar to a script or procedure. Action
62lists are implemented and expressed similarly to display lists in OpenGL. An action list encapsulates the specifics of a particular effect and allows complex
63effects to be treated as primitives like actions. An action list can be called from within another action list. </p> <p>Action lists also allow effects to be
64simulated much more efficiently because the entire set of actions is known at the start, and this knowledge can be used by the implementation of the Particle
65System API. For example, several actions can be applied to one set of particles before moving to the next set of particles, yielding better memory access
66patterns. Also, in future versions common sequences of actions might be detected and replaced by efficient code that handles all those actions in one pass. </p>
67<p>To take advantage of the up to 50% performance increase of action lists, you need to order your actions carefully. The performance advantage is achieved when
68two or more actions that can be combined are ordered next to each other in the list. Actions that create or delete particles (Vertex(), Source(), Sink*(), and
69Kill*()) and actions that reorder particles (Sort()) cannot be combined. All other actions can be combined to yield the performance advantage. See Effects.cpp
70for several examples of efficient ordering. </p>
71
72\subsection alistcalls Action List Calls
73<p>These calls create and operate on action lists, which are scripts of many actions to be applied together as a block to the current particle group. An empty
74action list is first created using GenActionLists, and is then filled or compiled by calling NewActionList, then calling Actions, then calling EndActionList.
75Once the action list is filled, it is run via CallActionList. Thus, an action list is sort of a higher-level action. Complex behaviors can be stored in an
76action list and then called later, even as part of another action list. Action lists cannot be edited. They can only be created or destroyed. To destroy an
77action list, call DeleteActionLists. When particles are created within a NewActionList / EndActionList block, they will receive attributes from the state that
78was current when the action list was created. </p> <p>When in immediate mode (not creating or calling an action list), particles are created with attributes
79from the current state. However, the time step length, dt, uses the value that is current when CallActionList is executed, not the value of dt when the action
80list was created. This allows dt to be modified without recompiling action lists. Maybe this isn't a good idea. If it should be the other way in the future, let
81me know. </p> <p><a href="ParticleLib/doc/html/class_p_a_p_i_1_1_p_context_action_list__t.html">List of Action List Calls </a></p>
82
83\subsection domains Domains
84<p>A Domain is a representation of a region of 3D space. For example, the Source action uses a domain to describe the volume in which a particle's random
85initial position will be. The Avoid(), Sink() and Bounce() actions use domains to describe a volume in space for particles to steer to avoid, die when they
86enter, or to bounce off, respectively. Domains are used as parameters to many functions within the API. See pDomain.h for more. </p>
87
88\subsection defparams Default Parameters
89<p>Many functions in the Particle System API take arguments with default values. This is a feature of C++. For example, in <code>Color(float r, float g, float
90b, float alpha = 1.0f) </code>, the parameter alpha has a default value of 1.0. This means that Color can be called with either three parameters or four. When
91called with three parameters, they are the values of r, g, and b; and alpha receives the default value, 1.0. Only parameters at the end of the parameter list
92can have default values. Likewise, when calling functions that have default values for parameters, all values specified will be applied to parameters starting
93with the left. This means that there is no way to specify a value for a parameter at the end of the list without specifying values for all parameters to its
94left. </p>

104\subsection sourcestate Source State Setting Calls
105<p>These calls dictate the properties of particles to be created by Source or Vertex. When particles are created within a NewActionList / EndActionList block,
106they will receive attributes from the state that was current when the action list was created. When in immediate mode (not creating or calling an action list),
107particles are created with attributes from the current state. </p> <p><a href="ParticleLib/doc/html/class_p_a_p_i_1_1_p_context_source_state__t.html">List of
108Source State Setting Calls </a></p>

110\subsection pgroup Particle Group Calls
111
112See PContextParticleGroup_t or \ref PContextParticleGroup_t huh.
113<p>A particle group is first created using GenParticleGroups, which will create a sequentially-numbered set of particle groups and return the identifying number
114of the first generated particle group. You specify which group is current using CurrentGroup. Unless otherwise stated, all other commands operate on the current
115particle group. The maximum number of particles in the group is specified using SetMaxParticles. The particle group is then acted upon using the <a
116href="ParticleLib/doc/html/class_p_a_p_i_1_1_p_context_actions__t.html">Actions </a>. </p> <p>After the actions have been applied, the particles are rendered.
117This is done at the same stage of the application's execution as drawing other geometry. To draw a particle group in OpenGL, the application calls GetParticles
118or GetParticlePointer functions to get the vertex data, then sends it to OpenGL. When a particle group is no longer needed, it is deleted using
119DeleteParticleGroups. </p> <p><a href="ParticleLib/doc/html/class_p_a_p_i_1_1_p_context_particle_group__t.html">List of Particle Group Calls </a></p>
